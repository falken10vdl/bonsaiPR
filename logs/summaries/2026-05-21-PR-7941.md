## Root Cause

The BonsaiPR `2605220101` build merges PRs in **descending** numeric order (highest first).
PR #7944 is merged before PR #7941 is attempted.

**PR #7941's sole commit:** [`2b9eecf0a6`](https://github.com/IfcOpenShell/IfcOpenShell/commit/2b9eecf0a6)
— "Fix extend_walls_to_underside ridge artifact"

**PR #7944's first commit:** [`06e364377d`](https://github.com/IfcOpenShell/IfcOpenShell/commit/06e364377d)
— exact same message, author, date, and diff as `2b9eecf0a6`, just a different SHA because
they branched from v0.8.0 independently.

PR #7944 then adds 6 more commits on top of that ridge fix, significantly evolving all
three conflicting files. When the build tries to merge PR #7941 after PR #7944, the
merge-base is v0.8.0, and PR #7941's patch no longer applies cleanly against the evolved
build tip.

**Conflicting files (all 3 changed by both PRs relative to the same v0.8.0 ancestor):**

- `src/bonsai/bonsai/bim/module/model/wall.py` — `ExtendWallsToUnderside._execute`: PR #7944 replaced the single-`slab` active-object approach with a multi-slab `get_selected_objects()` loop
- `src/bonsai/bonsai/core/model.py` — `extend_wall_to_slab`: PR #7944 rewrote the function to accept `slab_objs: list` and handle item mode
- `src/bonsai/bonsai/tool/model.py` — `clip_wall_to_slab`: PR #7944 added boolean tracking, `mark_manual_booleans`, slab connection

---

## Fix Strategy: Ancestry Merge (Option A)

PR #7941's content doesn't need to change — its ridge-fix logic is entirely subsumed by
PR #7944's first commit. The conflict is purely an ancestry problem.

**Before fix:**
- LCA of (build tip after PR #7944) and (PR #7941 branch) = v0.8.0
- PR #7941's patch targets v0.8.0 baseline — conflicts with PR #7944's evolved code

**After fix (merge PR #7944's tip into PR #7941):**
- LCA shifts to PR #7944's tip [`26bd75df31`](https://github.com/IfcOpenShell/IfcOpenShell/commit/26bd75df31)
- PR #7941 makes zero new changes above that LCA → trivially clean merge
