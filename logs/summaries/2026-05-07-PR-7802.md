## Root Cause

The BonsaiPR `2605071648` build merges PRs in **descending** numeric order (highest PR# first). PR #7924 is merged before PR #7802 is attempted.

**The conflicting file:** `src/bonsai/bonsai/core/root.py`, lines ~58–66

**Common ancestor ([`b4558f7f75`](https://github.com/IfcOpenShell/IfcOpenShell/commit/b4558f7f75), v0.8.0 base):**
```python
        # Only assign styles if element doesn't get them from material
        if not _has_material_styles(ifc, new):
            root.assign_body_styles(new, obj)
```

**After PR #7924 merges (commit [`aa3e38aca8`](https://github.com/IfcOpenShell/IfcOpenShell/commit/aa3e38aca8)) — build HEAD:**
```python
        # Only assign styles if element doesn't get them from material
        if not root.has_material_styles(new):
            root.assign_body_styles(new, obj)
```
(Refactored private function `_has_material_styles(ifc, new)` → public `root.has_material_styles(new)`)

**PR #7802's branch — what it contributes:**
```python
        # not sure what the purpose of this is, but removing it fixes wrong IfcStyledItems being assigned to
        # a duplicated object. Instead, copy_representation now properly copies IfcStyledItems directly.
        # root.assign_body_styles(new, obj)
```
(Removes the call entirely, as the fix copies IfcStyledItems directly in `copy_representation`)

**Why it conflicts:** Both sides change the same lines relative to the common ancestor — git's 3-way merge cannot determine which version should win.

---

## Why a Simple Commit Edit Doesn't Work

Even if PR #7802's commit is amended to reference `root.has_material_styles(new)` instead of `_has_material_styles(ifc, new)`, the 3-way merge still conflicts because:
- Merge-base is still `b4558f7f75`
- PR #7802's branch still changes lines 58–66 (to comment them out)
- The build (ours) still changes lines 58–66 (to the refactored form)
- Same conflict, different wording

---

## The Fix: Make `aa3e38aca8` an Ancestor of PR #7802's Branch

The solution is to merge commit [`aa3e38aca8`](https://github.com/IfcOpenShell/IfcOpenShell/commit/aa3e38aca8) directly into the PR #7802 branch. This changes the git ancestry:

**Before:** Merge-base of (build tip after PR#7924) and (PR#7802 branch) = `b4558f7f75`
**After:** Merge-base = `aa3e38aca8` (because it's now a literal ancestor of PR#7802)

When the build merges PR#7924 and then attempts PR#7802, the 3-way merge uses `aa3e38aca8` as the base. At that base, `core/root.py` already has `root.has_material_styles(new)`. The build (ours) doesn't change it further. Only PR#7802 (theirs) makes changes — removing the call. No conflict.

**Conflict during the fix merge itself** (in `core/root.py`): Resolved by keeping PR#7802's commented-out block (ours/HEAD side). The `root.has_material_styles` call from `aa3e38aca8` is intentionally discarded since PR#7802's whole point is to remove that code path.
