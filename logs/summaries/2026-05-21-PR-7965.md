## Which PRs conflict and why

Both PR #7965 and PR #8083 independently modified the same three files without knowledge of each other, both branched from `v0.8.0`:

| File | PR #7965 change | PR #8083 change |
|------|----------------|----------------|
| `Psets_BBIM_Annotation.ifc` | Added `IsManualDrawingReference` and `IsDocumentReference` as IFC entities **#34 and #35** to `EPset_Annotation` | Added `Separator`, `SuppressZeroFeet`, `IsOrdinate`, `Anchors`, `MeasureAxis`, `ForcePerpendicularToFace`, `LinePosition` as IFC entities **#34 through #40** to `BBIM_Dimension` |
| `drawing/prop.py` | Added `is_manual_reference: bpy.props.BoolProperty` after `type_name` | Added `force_perpendicular_to_face` and `line_position` properties after `type_name` |
| `drawing/workspace.py` | Added `ELEVATION`/`SECTION` UI row and changed `add_annotation()` → `add_annotation("INVOKE_DEFAULT")` | Added `DIMENSION_TYPES` UI row and replaced `hotkey_S_A` with parametric-dimension routing |

The IFC file conflict is the most concrete: both PRs allocate IDs `#34` and `#35` for entirely different properties. Git sees two branches simultaneously adding different content at IDs that collide.

## Why the chosen fix strategy works (git 3-way merge mechanics)

**Strategy: ancestry-merge** — `git merge origin/parametric_dimensions` into `inset_section_endpoints`.

Before the fix:
- **LCA** between build tip and `inset_section_endpoints` = some commit on `v0.8.0` (no `parametric_dimensions` in either's ancestry)
- git sees **both sides** adding content at the same locations → conflict

After the fix:
- `origin/parametric_dimensions` tip (`913c9c42ba`) is now an ancestor of `inset_section_endpoints` (via merge commit `738110bed7`)
- **LCA** between build tip and `inset_section_endpoints` = `913c9c42ba` (the parametric_dimensions tip, which is already in both branches)
- Build side shows: **no diff** relative to LCA (parametric_dimensions already in build)
- PR #7965 side shows: only its own new additions (the manual-reference properties at #41/#42, `is_manual_reference`, UI rows)
- git sees **only one side** making changes relative to the LCA → no conflict

## Files changed and how conflicts were resolved

### `src/bonsai/bonsai/bim/data/pset/Psets_BBIM_Annotation.ifc`

**Conflict:** Both PRs used IFC entity IDs `#34` and `#35`.

**Resolution:**
- Keep PR #8083's `#34`–`#40` unchanged
- Renumber PR #7965's `IsManualDrawingReference` from `#34` → **`#41`**
- Renumber PR #7965's `IsDocumentReference` from `#35` → **`#42`**
- Update `#1 EPset_Annotation` property list from `(…,#34,#35)` → `(…,#41,#42)`

Final state: IDs #34–#40 = PR #8083 (BBIM_Dimension props); #41–#42 = PR #7965 (EPset_Annotation props).

### `src/bonsai/bonsai/bim/module/drawing/prop.py`

**Conflict:** Both PRs inserted `bpy.props` declarations after `type_name` at the same location.

**Resolution:** Keep both sets of properties — PR #8083's `force_perpendicular_to_face` and `line_position` first, then PR #7965's `is_manual_reference`. No functional change to either.

### `src/bonsai/bonsai/bim/module/drawing/workspace.py`

Two conflict regions:

1. **After "Add annotation" UI row:** Both added an `if object_type in (...)` block. **Resolution:** Keep both — ELEVATION/SECTION block (PR #7965) first, then DIMENSION_TYPES block (PR #8083). Independent UI features.

2. **`hotkey_S_A`:** PR #8083 restructured the function to route dimension types to `draw_parametric_dimension`; PR #7965 added `"INVOKE_DEFAULT"` to `add_annotation()`. **Resolution:** Use PR #8083's routing structure with PR #7965's `"INVOKE_DEFAULT"` argument in the `elif` branch:
   ```python
   def hotkey_S_A(self):
       props = tool.Drawing.get_annotation_props()
       if props.object_type in self._PARAMETRIC_DIMENSION_TYPES:
           if bpy.ops.bim.draw_parametric_dimension.poll():
               bpy.ops.bim.draw_parametric_dimension("INVOKE_DEFAULT")
       elif bpy.ops.bim.add_annotation.poll():
           bpy.ops.bim.add_annotation("INVOKE_DEFAULT")
   ```
