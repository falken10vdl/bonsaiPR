## Which PRs conflict and why

Both PRs edit the **same region** of `BIM_PT_representations.draw()` in
[`src/bonsai/bonsai/bim/module/geometry/ui.py`](https://github.com/IfcOpenShell/IfcOpenShell/blob/93b80a615b/src/bonsai/bonsai/bim/module/geometry/ui.py),
the per-representation row in the representations panel:

- **#8200** inserts a new `RepresentationIdentifier` column label (between the `TargetView`
  and `RepresentationType` labels) and adds a header row above the loop.
- **#7916** replaces the plain `RepresentationType` label with a clickable
  `bim.select_by_representation_type` operator.

They are functionally independent (one adds a column, the other makes the Type column
clickable) but both rewrite the `row.label(text=representation["RepresentationType"])`
line and its neighbours, so git's 3-way merge sees both sides changing the same lines
relative to their common ancestor.

Build order is **ascending**, so #7916 (lower number) is merged into the build before
#8200 is attempted. #8200 merges cleanly against `v0.8.0` alone, hence the build status
"Merges cleanly with base. Skipped ŌĆō Conflict with other PRs."

The other PRs flagged by a naive `git merge-tree` scan (#7003, #7931, #7669, #7850) are
**false positives** from stale local merge-bases ŌĆö their conflicts are in files #8200
never touches (`core/root.py`, `model/wall.py`, `tool/blender.py`, `bim/__init__.py`).
`opening.py` auto-merges with #7003.

## Conflicting commit

[`f59942c453`](https://github.com/IfcOpenShell/IfcOpenShell/commit/f59942c453) ŌĆö the single

## Why the fix works (3-way merge mechanics)

`ancestry-merge`: `git merge f59942c453` into the `opening-template-on-type` branch.

- **Before:** in the build (`v0.8.0` + #7916 + ŌĆ” + #8200), the merge-base between the build
  tip and #8200 is `v0.8.0`. From that base, the build side carries #7916's `ui.py` rewrite
  and the #8200 side carries #8200's `ui.py` rewrite ŌĆö both touch the same lines ŌåÆ conflict.
- **After:** merging `f59942c453` makes it an **ancestor** of #8200. The build's merge-base
  with #8200 shifts from `v0.8.0` to `f59942c453`. Relative to that new base, #8200's only
  remaining `ui.py` delta is the added `RepresentationIdentifier` label + header row (the
  Type-operator change is now part of the base, shared by both sides). No region is
  contested ŌåÆ the build auto-merges cleanly.

## Files changed / how the conflict was resolved

Only `geometry/ui.py` required manual resolution. Kept **both** changes in the row:

```python
row.label(text=representation["ContextType"])
row.label(text=representation["ContextIdentifier"])
row.label(text=representation["TargetView"])
row.label(text=representation["RepresentationIdentifier"])   # #8200
op = row.operator(                                           # #7916
    "bim.select_by_representation_type",
    text=representation["RepresentationType"],
    emboss=False,
)
op.representation_type = representation["RepresentationType"]
```

The header row added by #8200 (above the loop) is unchanged. `geometry/operator.py`
auto-merged. No #8200 functional code changed ŌĆö only the textual co-location was resolved.
