## Conflict Files and Root Causes

### `drawing/operator.py`

**Root cause:** git LCA = `v0.8.0`, which has no `DrawParametricDimension`. Both the build
(via old pd commits) and new pd (via new pd commits) added `DrawParametricDimension` from
that empty base. The versions differ because new pd has 2 additional commits
([631881932d](https://github.com/IfcOpenShell/IfcOpenShell/commit/631881932d) — Fix
coplanar face snap, and
[76e390f820](https://github.com/IfcOpenShell/IfcOpenShell/commit/76e390f820) — Improve
SetDimensionAnchor snap) that the build's old pd does not. Git treated both sides as
"adding" the class from nothing and conflicted on the differing content.

**Conflict regions (all inside `DrawParametricDimension`):**
- `_compute_snap_cand_coplanar`: build missing `face_normal_world` key
- `_pt_in_obj_bbox`: build has old signature `(obj, pt_world, tol=1e-3)`, new pd has `(obj, pt_world, tol=1e-3, tol_z=None)` with Z-axis override
- `_compute_ifc_snap_candidate`: build missing hit_obj-first check and screen-space bbox filtering
- `_build_anchor_from_candidate`: build missing FACE snap_kind branch
- `_apply_hover_highlight`: build has older FACE draw handling

### `Psets_BBIM_Annotation.ifc`

**Root cause:** PR #7798 added `#41=IsManualDrawingReference` and `#42=IsDocumentReference`
to `EPset_Annotation`. New pd ends at `#40` (its own additions). With LCA = `v0.8.0`
(no `#34`–`#40`), git saw both sides adding entries from the same baseline. Build had
`#34`–`#42`; new pd had only `#34`–`#40` → conflict at `#41`.

### `drawing/prop.py`

**Root cause:** PR #7798 added `is_manual_reference: bpy.props.BoolProperty(...)`. New pd
doesn't have this. With LCA = `v0.8.0`, git saw build adding it, new pd not → conflict.

### `drawing/workspace.py`

**Root cause (two regions):**
1. PR #7798 added `is_manual_reference` UI block (ELEVATION/SECTION branch) — new pd
   didn't have it.
2. New pd has `bpy.ops.bim.add_annotation()` (no args). Build has
   `bpy.ops.bim.add_annotation("INVOKE_DEFAULT")` (set by previous conflict resolution
   [738110bed7](https://github.com/IfcOpenShell/IfcOpenShell/commit/738110bed7)).

## Why the Fix Works (git 3-way merge mechanics)

The two new commits on new pd (`631881932d`, `76e390f820`) **only touch `operator.py`**.
`prop.py`, `workspace.py`, and the pset file are byte-for-byte identical in new pd and
old pd (`913c9c42ba`).

Merging `913c9c42ba` into `parametric_dimensions` with `-s ours` (no content change,
just records the commit as a parent) shifts the LCA for the build's merge from `v0.8.0`
to `913c9c42ba`:

| File | LCA state | Build change from LCA | New pd change from LCA | 3-way result |
|------|-----------|-----------------------|------------------------|--------------|
| `operator.py` (DrawParametricDimension) | Old pd V1 | None (untouched by #7798/other PRs) | +2 new commits (V2) | Clean — only pd changed → take V2 |
| `prop.py` | No `is_manual_reference` | Added (from #7798) | None (= LCA) | Clean — take build's addition |
| `workspace.py` | No `is_manual_reference`; `add_annotation()` | Added UI block; changed to `add_annotation("INVOKE_DEFAULT")` | None (= LCA) | Clean — take all build changes |
| `Psets_BBIM_Annotation.ifc` | `#34`–`#40` | Added `#41`, `#42` | None (= LCA) | Clean — take build's additions |
