## Why the conflict occurs ŌĆö 3-way merge mechanics

The common ancestor (LCA) of the build (at the skip point `d95f949cf3`, after PR #7802 was
merged) and the general-mirroring branch was `7e96692764` (v0.8.0 tip at rebase time).

At that LCA, `FinishEditingDoor` had only:
```python
def _execute(self, context: bpy.types.Context) -> set[str]:
    return self._finish_targets(context)
```

**Build side** (changed from LCA): PR #7802 expanded `FinishEditingDoor` with the old general-mirroring
commits ŌĆö `finish_editing_door_on_object` instance method + `copy_door_params` instance
method + old-style `_execute` that loops over objects.

**general-mirroring side** (changed from LCA): Added `_finish_one` classmethod override,
`copy_door_params` classmethod, and kept `_execute -> _finish_targets` (correct mixin pattern).

Both sides changed the same region of the same class from the same base ŌåÆ conflict.

---

## Why the chosen fix works

The fix is `git merge -s ours 361955428c` on the general-mirroring branch. This:

1. Makes `361955428c` an **ancestor** of general-mirroring (via the merge's second parent)
2. Keeps general-mirroring's content **entirely unchanged** (the "ours" strategy discards the incoming content)
3. Shifts the **3-way merge LCA** when the build later merges general-mirroring

**New merge-base:** `361955428c` is now in the ancestry of both:
- The build (via `caaddd33a8`, the PR #7802 merge commit)
- general-mirroring (via the new ours-merge commit `cb500f4280`)

**At the new LCA (`361955428c`)**, `FinishEditingDoor` has the `finish_editing_door_on_object` form.

**From LCA to build (`d95f949cf3`):** No further changes were made to `FinishEditingDoor`.
(PR #7653 only added a material load loop to `AddDoor`, a different class ŌĆö it auto-merges.)

**From LCA to general-mirroring:** `_finish_one` classmethod replaces `finish_editing_door_on_object`.

Git sees only one side changed `FinishEditingDoor` ŌåÆ auto-merge succeeds with general-mirroring's `_finish_one` form winning.
