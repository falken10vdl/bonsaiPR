## Nature of the Conflict

Both PRs independently modify the same lines of `isolate_objects` relative to the common v0.8.0 base:

**Base at `d3f0ad03fb`:**
```python

bpy.ops.object.select_all(action="DESELECT")   # ← both PRs touch this line
```

**PR #8000's change:** Moves both `select_all` calls inside the `with` context blocks (operator context fix).

**PR #7744's change:** Adds `viewport_restricted` dict before the first `with` block, adds `hide_select_objs` collection in the loop, restores both after the second `with` block.

Both modifications overlap in the indented region between the two `with` blocks.

PR #7744 was authored by `theoryshaw` and branched from `61cfa48c2c` (a v0.8.0 commit from ~March 2026). PR #8000 was branched much later from `4adaf0d61f`. Because PR #7744's branch point is so much older, git's 3-way LCA in the build is `61cfa48c2c`, causing git to see ALL of v0.8.0's `blender.py` changes (since that older point) plus PR #8000's changes as "ours" vs PR #7744's single `isolate_objects` commit as "theirs".

## Why Rebase (not ancestry-merge)

**Option A (ancestry-merge)** was considered: merging `d72c400756` into PR #7744's branch would shift the LCA and eliminate the v0.8.0 accumulation noise — but the `isolate_objects` region would still conflict, because PR #8000 and PR #7744 both modify the same lines. The content of PR #7744 genuinely needed updating to incorporate PR #8000's context fix. → **Content must change → rebase.**

**Option B (rebase):** Rebasing PR #7744 onto `d72c400756` (PR #8000's HEAD):
- Shifts PR #7744's ancestry past all of PR #8000's changes
- When the build merges PR #8000 first (descending order), then PR #7744, the LCA is `d72c400756`
- From that LCA: "ours" (build state) makes no further changes to `isolate_objects` (confirmed: no other merged PR between #7744 and #8000 touches `blender.py`; the two v0.8.0 commits between `4adaf0d61f` and `d3f0ad03fb` that touch `blender.py` — `233cc344fa` and `2143262883` — do not change `isolate_objects`)
- "Theirs" (rebased PR #7744) only adds the `viewport_restricted` + `hide_select` logic → **clean merge**

## Conflict Resolution During Rebase

During `git rebase d72c4007562`, one conflict arose in `blender.py`:

**Conflict 1** — first `with` block end:
- HEAD (PR #8000): `select_all` inside the `with` block
- PR #7744: ends `with` block, then `select_all` outside, then `hide_select_objs = []`
- **Resolution:** Keep `select_all` inside the `with` block; add `hide_select_objs = []` after the `with` block

**Conflict 2** — second `with` block end:
- HEAD (PR #8000): `select_all` inside the `with` block
- PR #7744: after `with` block, adds `for obj in hide_select_objs:`, `for obj, was_hidden in viewport_restricted.items():`, then standalone `select_all`
- **Resolution:** Keep `select_all` inside the `with` block; add `hide_select_objs` restore and `viewport_restricted` restore after the `with` block; drop the redundant standalone `select_all`

**Combined result:**
```python
override = cls.get_viewport_context()

# Save H-key hide state for globally-restricted objects so we don't change it.
viewport_restricted = {obj: obj.hide_get() for obj in bpy.context.view_layer.objects if obj.hide_viewport}

with bpy.context.temp_override(**override):
    bpy.ops.object.hide_view_clear(select=False)
    bpy.ops.object.select_all(action="DESELECT")   # PR #8000: inside with context

hide_select_objs = []
for obj in objs:
    if obj.hide_select:
        hide_select_objs.append(obj)
        obj.hide_select = False
    obj.select_set(True)
with bpy.context.temp_override(**override):
    bpy.ops.object.hide_view_set(unselected=True)
    bpy.ops.object.select_all(action="DESELECT")   # PR #8000: inside with context

for obj in hide_select_objs:
    obj.hide_select = True

for obj, was_hidden in viewport_restricted.items():
    obj.hide_set(was_hidden)

for name in previously_selected:
    ...
```
